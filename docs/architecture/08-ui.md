# 08 — UI (kacho-ui integration)

> **Forward-looking.** UI строится отдельно (репо `kacho-ui` — Vite + React SPA).
> Этот документ фиксирует **намеренный дизайн** compute-views, чтобы при
> реализации не было расхождений с принципами проекта. Принцип 6 (workspace
> `CLAUDE.md`): для каждого модуля — UI, встроенный в `kacho-ui`; generic-механизм
> CRUD-страниц, **не копипаста под каждый ресурс** (memory feedback «Fix
> systemically»).

## Размещение

Compute-views живут в `kacho-ui/src/pages/compute/`:

```
kacho-ui/src/pages/compute/
├── instances/   — list + detail + create-wizard для Instance
├── disks/       — list + detail + create-wizard для Disk
├── images/      — list + detail + create-wizard для Image
└── snapshots/   — list + detail + create-wizard для Snapshot
```

DiskType и Zone — **не отдельные страницы** (read-only справочники); используются
как dropdown-источники в create-формах (`/compute/v1/diskTypes`,
`/compute/v1/zones`) и опционально как admin-таблицы под флагом (через
`Internal*` REST на cluster-internal listener).

## Транспорт

UI ходит **только** в REST api-gateway (`/compute/v1/...`) — никогда напрямую в
gRPC `:9090`/`:9091` сервиса. В dev — port-forward `svc/api-gateway 18080:8080`.
Базовый URL берётся из конфигурации SPA (env / runtime config). UI **не
использует** `InternalWatchService` (server-stream) — реактивность через
**polling** (см. ниже).

| Действие в UI | REST endpoint |
|---|---|
| Список ВМ folder'а | `GET /compute/v1/instances?folderId=&pageSize=&pageToken=&filter=` |
| Деталь ВМ | `GET /compute/v1/instances/{id}?view=FULL` (FULL — чтобы показать metadata) |
| Создать ВМ | `POST /compute/v1/instances` → `Operation` (epd...) |
| Start / Stop / Restart ВМ | `POST /compute/v1/instances/{id}:start` / `:stop` / `:restart` → `Operation` |
| Attach / Detach disk | `POST /compute/v1/instances/{id}:attachDisk` / `:detachDisk` → `Operation` |
| Add / Remove NAT | `POST /compute/v1/instances/{id}/addOneToOneNat` / `/removeOneToOneNat` → `Operation` |
| Update / Delete ВМ | `PATCH` / `DELETE /compute/v1/instances/{id}` → `Operation` |
| Serial port output | `GET /compute/v1/instances/{id}:serialPortOutput?port=1` (sync, не операция) |
| Список / деталь / создать / обновить / удалить Disk | `GET\|POST\|PATCH\|DELETE /compute/v1/disks[/{id}]`; `POST .../{id}:relocate` |
| Список / деталь / создать / обновить / удалить Image | `GET\|POST\|PATCH\|DELETE /compute/v1/images[/{id}]`; `GET /compute/v1/images:latestByFamily?folderId=&family=` |
| Список / деталь / создать / обновить / удалить Snapshot | `GET\|POST\|PATCH\|DELETE /compute/v1/snapshots[/{id}]` |
| Опросить операцию | `GET /operations/{operationId}` (БЕЗ `/compute/v1/`; api-gateway opsproxy роутит по prefix `epd...`) |
| Dropdown-источники для create-форм | `GET /compute/v1/diskTypes`, `GET /compute/v1/zones` |
| (admin, опционально) seed справочников | `POST\|PATCH\|DELETE /compute/v1/diskTypes[/{id}]` / `/zones[/{id}]` — только на cluster-internal listener |

## Generic CRUD-page механизм

Один параметризуемый набор компонентов на все 4 ресурса (как memory feedback
требует — не копипаста):

- **`<ResourceListPage resource="...">`** — таблица + пагинация (cursor:
  `nextPageToken`) + поиск по `name` (filter `name="<v>"`) + кнопка «Создать» +
  per-row actions. Колонки и фильтр-поля декларируются дескриптором ресурса (см.
  ниже).
- **`<ResourceDetailPage resource="..." id="...">`** — карточка с полями ресурса +
  таб'ы (Overview / Operations history через `GET .../{id}/operations` / для
  Instance — Disks, Network interfaces, Serial console) + per-resource actions
  (Edit / Delete + lifecycle-actions).
- **`<ResourceCreateWizard resource="...">`** — форма из дескриптора; для
  Instance — многошаговый (Image/Disk source → Resources/Platform → Network →
  Metadata/SSH → Review).
- **`<OperationToast operationId="...">`** — toast/banner с прогрессом операции,
  поллит `GET /operations/{id}`.

**Дескриптор ресурса** (`resourceDescriptors.compute.ts`) — единый источник:
поля и их типы (text / number-bytes / labels-editor / enum-select / disk-type-select
/ zone-select / subnet-select(cross-service VPC) / sg-multi-select / source-oneof),
immutable-флаги (для grey-out в Edit-форме), валидационные regex'ы (зеркаль
proto-pattern: `NameCompute` — lowercase + digits + hyphens + underscore), список
list-колонок, filter-whitelist, доступные actions. Меняется один дескриптор —
обновляются все три страницы ресурса.

## Polling для Operation completion (вместо Watch)

С Phase 1.0 Watch RPC выкинут (workspace `CLAUDE.md`); UI реактивен через polling:

1. Мутирующий вызов (`POST /compute/v1/instances` и т.п.) возвращает `Operation
   {id: epd..., done: false}`.
2. UI кладёт `id` в очередь активных операций, показывает `<OperationToast>` с
   spinner'ом.
3. Каждые ~1.5-3 сек: `GET /operations/{id}`. Пока `done=false` — продолжаем.
4. `done=true` + `response` → success: распарсить `response` (Any → Instance /
   Disk / Empty), показать toast «Готово», обновить список/деталь (re-fetch).
5. `done=true` + `error` → fail: показать `error.code` + `error.message`
   (verbatim YC текст — уже без internal-обёртки).
6. Для общего состояния списков — фоновый re-fetch list каждые ~5 сек, пока
   страница открыта (опционально, или ручной refresh).

## Per-resource UI specifics

### Instances

- **List**: колонки id / name / status (badge: RUNNING=зелёный, STOPPED=серый,
  переходные — pulse, ERROR/CRASHED=красный) / zone_id / platform_id / cores ×
  memory / created_at. filter `name=`. Per-row: Start/Stop/Restart (enabled по
  текущему status — см. `03-instance-lifecycle.md`), Open, Delete.
- **Detail**: Overview (все поля); таб **Disks** — список `boot_disk` +
  `secondary_disks` (mode, device_name, auto_delete, → ссылка на Disk-detail);
  кнопки **Attach disk** (modal: выбрать существующий Disk same-zone READY или
  создать новый inline; mode, device_name, auto_delete) и **Detach disk**
  (per-secondary-disk; boot disk — disabled). Таб **Network interfaces** — список
  NIC (index, subnet_id → ссылка в VPC-UI, primary_v4_address, NAT external IP,
  security_group_ids); кнопки Add/Remove NAT, Update NIC (subnet/SG/static-IP).
  Таб **Serial console** — `GET .../{id}:serialPortOutput?port=1` (read-only
  textarea, refresh-кнопка; синтетический текст в Kachō). Таб **Operations** —
  история через `GET .../{id}/operations`. Actions: Start/Stop/Restart, Update
  (resources/platform — disabled пока не STOPPED, с подсказкой «Остановите ВМ»),
  Update metadata (key-value editor → `POST .../updateMetadata`), Delete.
- **Create wizard** (multistep): (1) Boot disk — выбрать Image
  (`/compute/v1/images?folderId=` или `:latestByFamily`) / Snapshot / существующий
  Disk, либо disk_spec (type_id из `/compute/v1/diskTypes`, size, block_size,
  auto_delete); (2) Resources & Platform — platform_id select → динамически
  ограничивает cores/memory/core_fraction/gpus (per-platform таблица — UI
  отзеркаливает `internal/service/platforms.go` или дёргает справочник, если
  такой будет); (3) Network — ⚠️ **на паузе**: `Instance.Create` больше не создаёт
  и не привязывает NIC (auto-NIC материализация удалена в `KAC-266`; правильная
  сетевая модель — будущая переделка), поэтому network-шаг wizard'а неактивен до
  возврата сетевой модели; (4) Metadata & SSH — key-value editor (включая
  `ssh-keys`), `serial_port_settings`, `service_account_id`; (5) Review →
  `POST /compute/v1/instances`.

### Disks

- **List**: id / name / status (CREATING/READY/ERROR/DELETING badge) / type_id /
  zone_id / size / `instance_ids` (если attached — бейдж «attached», нельзя
  Delete). Per-row: Open, Delete (disabled если attached → tooltip «диск
  подключён к ВМ»), Create snapshot (быстрый `POST /compute/v1/snapshots {disk_id}`).
- **Detail**: Overview + source (source_image_id / source_snapshot_id → ссылка);
  Actions: Edit (name/description/labels/size — size только увеличение, immutable
  поля grey-out), Relocate (если не attached), Delete, Create snapshot.
- **Create**: folder, zone_id (select), type_id (select `/compute/v1/diskTypes`),
  size (bytes, helper «GiB/TiB»), source — oneof none / image_id / snapshot_id,
  block_size (default 4096), labels. `kms_key_id` — задизейблено / скрыто
  (`blocked:kacho-kms`).

### Images

- **List**: id / name / family / status / os.type (LINUX/WINDOWS icon) /
  min_disk_size / created_at. filter `name=`. Per-row: Open, Delete, «Use →
  create disk».
- **Detail**: Overview + source (source_image_id / source_disk_id /
  source_snapshot_id / source_uri); Actions: Edit (name/description/labels/
  min_disk_size; family/os/product_ids/pooled — immutable grey-out), Delete.
- **Create**: folder, name, family, source — oneof image_id / disk_id /
  snapshot_id / uri (для uri — текстовое поле «pre-signed URL»; в Kachō download
  мгновенный), os.type select, min_disk_size, labels. «GetLatestByFamily» — отдельный
  поиск-виджет (folder + family → результат).

### Snapshots

- **List**: id / name / source_disk_id (→ ссылка) / status / disk_size /
  storage_size / created_at. filter `name=`. Per-row: Open, Delete, «Restore →
  create disk».
- **Detail**: Overview; Actions: Edit (name/description/labels; source_disk_id/
  disk_size/storage_size immutable grey-out), Delete, Restore-to-disk.
- **Create**: folder, disk_id (select из READY-дисков folder'а), name, labels.

## Cross-service навигация

- Instance NIC → ссылка на VPC-UI (`/vpc/v1/subnets/{subnet_id}`,
  `/vpc/v1/securityGroups/{sg_id}`, `/vpc/v1/addresses/{address_id}`).
- Disk source / Snapshot source / Image source — внутри compute-UI (та же БД).
- project picker — общий компонент (kacho-iam `/iam/v1/projects`; ранее
  resource-manager folders — заменён в KAC-124), один для всех модулей.

## Что НЕ делает UI

- Не показывает `Internal*` admin-функции на external-доступном UI (только
  internal admin-режим под флагом для seed справочников).
- Не использует Watch RPC (polling-only).
- Не дублирует бизнес-валидацию backend'а — клиентская валидация (proto-regex
  зеркаль) для UX, но source of truth — backend (`InvalidArgument` от него
  показывается как есть).
- Не реализует placement-group / host-group / gpu-cluster / filesystem UI (proto
  vendored, ресурсы не реализованы — соответствующие поля скрыты / задизейблены).
