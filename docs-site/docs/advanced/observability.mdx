# Наблюдаемость

kacho-compute инструментирован для эксплуатации в кластере: метрики, health-пробы,
структурированные логи и наблюдаемость асинхронных операций. Эта страница — обзор того, что
доступно оператору.

## Метрики

Сервис экспонирует метрики в формате Prometheus на HTTP-эндпоинте
(`KACHO_COMPUTE_METRICS_ADDR`). Помимо стандартных gRPC-метрик (частота, латентность,
коды ответов по методам), полезно отслеживать:

| Область | Что показывает |
|---|---|
| gRPC per-method | RPS, латентность p50/p95/p99, распределение gRPC-кодов |
| Операции (LRO) | число созданных / завершённых / упавших операций |
| Outbox / drainer | размер очереди, лаг доставки, повторные попытки регистрации owner-tuple |
| Пул БД | занятые/свободные соединения pgx |
| Watch-стримы | число активных стримов `InternalWatchService` |

## Health / readiness

Health-пробы (`internal/observability/health`) отражают готовность зависимостей: доступность
базы `kacho_compute` и, при `REQUIRE_IAM`, доступность kacho-iam на старте (boot-gate,
fail-fast). Readiness-проба используется для управления трафиком в оркестраторе.

## Логи

Логи структурированные (slog); в проде текст ошибок БД наружу **не** выводится в API (клиент
получает фиксированный `INTERNAL "internal database error"`), но в логах сохраняется контекст
для диагностики. Не логируйте секреты.

## Наблюдаемость операций

Асинхронные операции наблюдаемы через API: `ListOperations` по ресурсу показывает историю LRO,
а `OperationService.Get` — текущее состояние. Для диагностики «зависших» Create/Start/AttachDisk
это первичный инструмент — до анализа метрик и логов.

:::note «Задеплоено» ≠ «работает»
Rollout, Running-поды и применённые миграции — не доказательство работоспособности. Ряд классов
проблем (потеря principal в worker'е, расхождение миграций, недоступность peer-канала) видны
только на живом стеке через end-to-end проверку (запрос через api-gateway аутентифицированно).
Проверяйте реальным запросом, а не только статусом подов.
:::

## Диагностика частых ситуаций

| Симптом | Куда смотреть |
|---|---|
| Мутация падает `UNAVAILABLE` | Доступность peer-сервиса (iam/vpc/geo) на request-path |
| Ресурс не виден в `List` сразу после Create | Пропагация authz-tuple (~секунды) — поллинг, не мгновенный assert |
| Операция «зависла» `done: false` | `ListOperations` ресурса; логи worker'а; состояние drainer outbox |
| `INTERNAL "internal database error"` | Логи (текст SQL там); наличие объектов схемы (`to_regclass`) |
