# Особенности дизайна Kachō Compute

Kachō Compute спроектирован под задачу в чистой форме — это собственный продукт со своими
конвенциями. Ниже — осознанные дизайн-решения и объяснение «почему так»: они помогают понять
поведение API там, где оно может показаться неочевидным.

## Решения и обоснования

| Решение | Почему так |
|---|---|
| **Control-plane без data plane** | Сервис управляет намерением и состоянием (API, валидация, жизненный цикл, authz). Реального гипервизора нет: статус переходит детерминированной state-машиной, содержимое диска не материализуется, «загрузка» образа мгновенна, вывод serial-порта синтетический. Это позволяет сосредоточить контракт на управлении, отделив его от исполнения. |
| **Все мутации — асинхронные (`Operation`)** | Единый устойчивый контракт: клиент не блокируется, повторный опрос идемпотентен, запись об операции переживает рестарт. Синхронный возврат ресурса из мутации намеренно исключён. |
| **Instance создаётся без авто-NIC** | Жизненный цикл инстанса и сетевая привязка разделены: `Create` не провижинит сетевые интерфейсы, они подключаются отдельным действием `AttachNetworkInterface`. Это делает управление сетью инстанса явным и независимым, а сам NIC — first-class ресурсом домена kacho-vpc. |
| **Источник диска (image/snapshot) — не FK** | Диск хранит id источника, но образ/снимок можно удалить независимо. Так снимки и образы остаются самостоятельными ресурсами, а не «заложниками» созданных из них дисков; consumer graceful переживает dangling-ref. |
| **Присоединённый диск защищён FK RESTRICT** | Удаление диска, который числится в инстансе, отклоняется на уровне БД (`"The disk is being used"`) — инвариант выражен DB-конструкцией, а не software-проверкой (защита от гонок). |
| **Разные верхние пределы размера диска Create vs Update** | При `Create` допускается больший максимум, при `Update` — меньший; размер можно только увеличивать. Это осознанное ограничение операции расширения. |
| **DiskType — read-only на публичном API** | Каталог типов дисков задаёт администратор платформы; tenant только читает. Управление каталогом — admin-функция на internal-порту (`InternalDiskTypeService`), не на публичной поверхности. |
| **Geography (Region/Zone) — во внешнем домене** | Зоны и регионы — топологический leaf-домен kacho-geo. Compute не владеет ими, а валидирует `zoneId` вызовом `ZoneService.Get`. Это убирает дублирование топологии между сервисами. |
| **Общие ID-префиксы (`epd`, `fd8`)** | Instance/Disk делят `epd`, Image/Snapshot — `fd8`: группировка по домену внутри сервиса. Все Compute-операции получают `epd` — по нему api-gateway маршрутизирует `OperationService.Get` в единый backend. |
| **Placement скрыт из публичного Instance** | Привязка инстанса к физическому хосту/host-группе — инфра-чувствительные данные (разведка для lateral movement). Они исключены из публичной проекции `Instance` и доступны, при необходимости, только на internal-проекции. |

## Имя ресурса — lowercase-only

Имена ресурсов Compute подчиняются lowercase-контракту
(`^([a-z]([-_a-z0-9]{0,61}[a-z0-9])?)?$`): строчные буквы, цифры, дефис, подчёркивание. Это
осознанно строже, чем permissive-имена в некоторых других доменах Kachō — единообразие имён в
рамках домена важнее гибкости регистра.

## Возможности в развитии

Часть операций из proto-контракта в текущем control-plane ограничена или пока не реализована
(например, cross-zone Relocate дисков и отдельные интеграции хранилищ). Публичный API
документирует **фактически работающее** поведение; расширения появляются по мере развития
сервиса и отражаются в этой документации.

:::note Тексты ошибок — часть контракта
Формулировки ошибок (`"Instance must be stopped"`, `"Instance is not running"`, `"The disk is
being used"`, `"Disk size can only be increased"`) стабильны и являются частью контракта —
клиенты вправе на них полагаться. Меняются только осознанно, через тикет.
:::
